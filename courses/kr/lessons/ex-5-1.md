> The `getint` from §5.2 considers `+` or `-` to be the start of a number even when no digits follow - which leaves the buffer in a strange state on, say, `+a`. Rewrite `getint` so that an isolated `+`/`-` is treated as if no number were present and is pushed back along with anything else.

## Approach

After reading the optional sign, peek at the next character. If it's not a digit, push everything back - both the sign and the non-digit - and report "no number".

```c:starter
#include <stdio.h>
#include <ctype.h>

static int pushback;
static int has_pb = 0;

int getch(void) {
    if (has_pb) { has_pb = 0; return pushback; }
    return getchar();
}
void ungetch(int c) {
    if (has_pb) { fprintf(stderr, "ungetch: full\n"); return; }
    pushback = c; has_pb = 1;
}

int getint(int *pn);

int main(void) {
    int v;
    int rc;
    while ((rc = getint(&v)) != EOF) {
        if (rc == 0) printf("non-number, skipping\n");
        else         printf("got: %d\n", v);
    }
    return 0;
}

int getint(int *pn) {
    int c, sign;
    while (isspace(c = getch()))
        ;
    if (!isdigit(c) && c != EOF && c != '+' && c != '-') {
        ungetch(c);
        return 0;             /* not a number */
    }
    sign = (c == '-') ? -1 : 1;
    if (c == '+' || c == '-') {
        int next = getch();
        if (!isdigit(next)) {
            /* push back BOTH the sign and the non-digit */
            if (next != EOF) ungetch(next);
            ungetch(c);
            return 0;
        }
        c = next;
    }
    for (*pn = 0; isdigit(c); c = getch())
        *pn = 10 * (*pn) + (c - '0');
    *pn *= sign;
    if (c != EOF)
        ungetch(c);
    return c;
}
```

```output
(awaits input - try '+a 42 -7 ?')
```

The expected behaviour on the input `+a 42 -7 ?`:
- `+a` → non-number; push back both, the next call sees `+a` (still). The driver then skips one char or breaks the loop.
- `42` → got 42.
- `-7` → got -7.
- `?` → non-number.

A robust driver would treat the "non-number" return as a signal to advance one character and retry; otherwise an isolated `+` would cause infinite looping.

## Notes from the author

- The two-pushback case (the sign AND the non-digit) is why ex-4-8's single-pushback simplification doesn't suffice here. We need at least two slots, or the slightly more careful structure above.
- The "look ahead two characters" pattern is unavoidable for tokenisers that have to distinguish `+5` (number with explicit sign) from `+x` (unary plus operator). All real-world parsers handle this somewhere.
- Returning a tri-state (`EOF`, no number, number) via a single `int` is awkward C - the function uses the return value to mean "the next character", with `0` reserved as "no number". A modern API would split: `bool getint(int *out, bool *was_number)`. K&R's version is compact but harder to use correctly.
