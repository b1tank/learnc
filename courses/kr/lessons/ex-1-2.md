> **Exercise 1-2.** The C language defines a small set of *escape sequences* - `\n`, `\t`, `\\`, `\"`, `\0`, etc. What happens when the string passed to `printf` contains a backslash followed by some other character, like `\q` or `\z`? Does the compiler warn? Does the runtime do something useful? Find out empirically.

## Approach

Three things can happen to a backslash-followed-by-X inside a string literal:

1. **X is a recognised escape** - the compiler replaces the two-character sequence with the single byte it represents (`\n` → newline byte `0x0A`).
2. **X is an unrecognised character** - the C standard says the behaviour is *implementation-defined*. In practice every compiler I've used warns ("unknown escape sequence `\q`") and then keeps the *X* in the output, dropping the backslash.
3. **X is a digit** - `\nnn` is an octal escape (up to three octal digits). `\033` is ESC. This is *not* an unknown escape; it's a fully defined feature you'll meet in chapter 2.

Write a program that prints several backslash sequences and see what comes out.

```c:starter
#include <stdio.h>

int main(void) {
    printf("recognised: [\n] [\t] [\\] [\"]\n");
    printf("unknown:    [\q] [\z] [\m]\n");
    printf("octal:      [\101] [\102] [\103]\n");
    return 0;
}
```

```output
recognised: [
] [	] [\] ["]
unknown:    [q] [z] [m]
octal:      [A] [B] [C]
```

## Solution walkthrough

- The recognised escapes do what you expect: newline, tab, literal backslash, literal quote.
- The "unknown" backslash sequences (`\q`, `\z`, `\m`) produce a compiler warning during build (something like *unknown escape sequence: '\q'*) but the program still links and runs. At runtime the backslash is silently dropped - `printf` writes just `q`, `z`, `m`. This is the behaviour of every mainstream compiler (gcc, clang, MSVC).
- The octal escapes `\101 \102 \103` are 65, 66, 67 in decimal - the ASCII codes for `A`, `B`, `C`.

## Notes from the author

- The lesson here is twofold: (a) the small set of *defined* escapes is worth memorising (`\n \t \r \\ \" \' \0 \a \b \f \v`), and (b) leaning on unknown escapes is a bug magnet - your "literal `\q`" today becomes a real escape tomorrow if C ever expands the set.
- The octal example is a sneaky preview of §2.3 (Constants). I left it in because it lets you produce non-printable characters portably (e.g. `\033[31m` is ANSI red).
- Modern compilers under `-Werror=unknown-escape-sequence` will refuse to build this program. That's a *good* default to use on real code.
