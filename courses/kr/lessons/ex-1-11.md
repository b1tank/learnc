> **Exercise 1-11.** The word-counting program in §1.5 has a tiny state machine. How would you *test* it? What inputs are most likely to expose bugs?

## Approach

No code to write - this exercise is the first time the book points at testing as a skill. Open the §1.5 starter alongside this lesson and walk through these categories.

### A test plan

1. **Empty input.** Zero lines, zero words, zero characters. Boundary case for every counter.
2. **Whitespace only.** Spaces, tabs, newlines, but no actual words. Should report `>0` characters but `0` words and `≥0` lines.
3. **One word, no surrounding whitespace, no trailing newline.** Tests the `OUT → IN` transition exactly once and what happens when EOF arrives mid-word.
4. **One word with leading whitespace.** Should still count as one word.
5. **Multiple words separated by single spaces.** The happy path.
6. **Words separated by multiple spaces, mixed with tabs and newlines.** The state machine should treat any whitespace run as a single separator.
7. **A line of length exactly 1.** A single character followed by `\n`. Off-by-one detector.
8. **A line with no terminating newline.** Final word counted? Line counted?
9. **Very long lines / very long input** - does anything overflow? (For this program with `int` counters, you'd need billions of characters to wrap around. But the muscle is what matters.)
10. **Non-ASCII bytes.** Bytes ≥ 128. Does the program still treat them as part of a word? Should it?

```c:starter
/* §1.5 word counter - load into the editor and try the inputs above */
#include <stdio.h>

#define IN  1
#define OUT 0

int main(void) {
    int c, nl = 0, nw = 0, nc = 0, state = OUT;
    while ((c = getchar()) != EOF) {
        ++nc;
        if (c == '\n') ++nl;
        if (c == ' ' || c == '\n' || c == '\t')
            state = OUT;
        else if (state == OUT) {
            state = IN;
            ++nw;
        }
    }
    printf("%d lines, %d words, %d chars\n", nl, nw, nc);
    return 0;
}
```

## Notes from the author

- Bugs hide in the *transitions* of state machines, not in the steady states. So focus tests on boundaries: empty/non-empty, last-character/first-character, alone/in-a-run.
- A widely useful habit: write a tiny script (or shell one-liner) that pipes a fixed input into your program and `diff`s the output against an expected fixture file. CI is just that, scaled.
- The K&R word counter has a known subtle bug: on input that doesn't end in a newline but ends *inside* a word, the word is still counted (because we increment `nw` on the `OUT → IN` transition, not on the `IN → OUT` exit). That happens to be the right behaviour for most use-cases - but it's an asymmetry worth being conscious of.
- Stretch: when you reach chapter 8, add a `--unit-test` mode that runs a battery of fixed inputs and asserts the expected counts. That's the same architecture real `wc` clones use.
