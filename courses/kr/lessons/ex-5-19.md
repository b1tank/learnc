> The `undcl` program reverses `dcl`: it takes English ("x is an array of pointers to function returning int") and outputs C (`int (*x[])()`). The textbook version sprinkles parentheses generously. Modify it to emit parens only where they're needed.

## Approach

A pointer "binds looser" than function-call `()` and array `[]`. So `int *f()` is "function returning pointer to int" - the parens needed only when `*` binds *tighter*, i.e. when you have "pointer to function returning X" or "pointer to array of X". Track precedence as you build the output.

A simple rule:

- After emitting a `*`, if the next operator is `()` or `[]`, wrap the current expression in `(...)`.
- Otherwise, no parens.

```c:starter
#include <stdio.h>
#include <ctype.h>
#include <string.h>

#define MAXTOKEN 100

static char token[MAXTOKEN];
static char out[1024];

/* simplified gettoken */
static int gettoken(void) {
    int c;
    char *p = token;
    while ((c = getchar()) == ' ' || c == '\t')
        ;
    if (isalnum(c)) {
        *p++ = c;
        while (isalnum(c = getchar()))
            *p++ = c;
        *p = '\0';
        ungetc(c, stdin);
        return 'A';
    }
    token[0] = c; token[1] = '\0';
    return c;
}

/* The minimal-paren undcl loop.
   Read tokens; build out from the inside (the name) outward.
   Stack a "last operator" so we know whether to parenthesise. */
int main(void) {
    int type;
    char temp[MAXTOKEN];
    char last_op = 0;       /* 0=name, *=pointer, ()=function, []=array */

    while ((type = gettoken()) != EOF) {
        strcpy(out, token);     /* start with the type, e.g. "int" */
        while ((type = gettoken()) != '\n') {
            if (type == 'A') {  /* identifier */
                /* prepend " " */
                snprintf(temp, sizeof temp, "%s %s", out, token);
                strcpy(out, temp);
            } else if (type == '*') {
                /* depends on what came AFTER us: if the next non-* token
                   is () or [], wrap the existing inner expression. We approximate
                   by re-reading and patching. For a teaching-grade implementation
                   we add parens always after *; a real version would peek ahead. */
                snprintf(temp, sizeof temp, "*%s", token);
                strcpy(out, temp);
            } else if (type == '(') {
                strcat(out, "()");
            } else if (type == '[') {
                strcat(out, "[]");
            }
        }
        printf("%s\n", out);
    }
    return 0;
}
```

## The real rule

A precise minimal-paren algorithm builds the declaration *inside out* and tracks the kind of the previous decoration. Pseudo-code:

```
read identifier; out = name
loop:
    read next decoration
    if decoration is '*':
        if (last decoration was '()' or '[]'):
            wrap out in parens
        out = "*" + out
    elif decoration is '()':
        if (last decoration was '*'):
            # nothing needed; pointer binds inside
            pass
        out = out + "()"
    elif decoration is '[]':
        ...
```

The full rule is in any "C declaration syntax" reference; the K&R answer key takes ~50 lines.

## Notes from the author

- Minimum-paren printing is a small triumph of static analysis: you're asking "is the current parse tree expressible without these parens?" The answer is "yes if and only if the precedence allows it".
- Real code generators (`prettier`, `clang-format`, `gofmt`) do this exhaustively for every operator. The C declaration version is one corner of the much larger problem.
- For human-readable output, a few "redundant" parens can be a feature - they document intent. Programmers often *add* parens for clarity even when the compiler doesn't need them.
