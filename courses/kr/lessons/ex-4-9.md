> The simple `getch`/`ungetch` from ex-4-8 can't push back EOF, because EOF is used as the "buffer empty" sentinel. Make the pair work correctly if EOF is pushed back.

## Approach

Use a separate flag, not a sentinel value. A `bool` (or `int` flag) says "is there a pushed-back character?", and a `int` slot holds the character (which may be EOF).

```c:starter
#include <stdio.h>
#include <stdbool.h>

static int   pushback;
static bool  has_pushback = false;

int getch(void) {
    if (has_pushback) {
        has_pushback = false;
        return pushback;
    }
    return getchar();
}

void ungetch(int c) {
    if (has_pushback) {
        fprintf(stderr, "ungetch: only one char allowed\n");
        return;
    }
    pushback     = c;
    has_pushback = true;
}

int main(void) {
    /* pretend we read EOF, want to push it back so the next getch sees it */
    ungetch(EOF);
    int c = getch();
    if (c == EOF) printf("got EOF back\n");
    else          printf("got '%c'\n", c);

    /* push back a normal char */
    ungetch('A');
    c = getch();
    printf("got '%c'\n", c);

    return 0;
}
```

```output
got EOF back
got 'A'
```

## Why this matters

In a real parser, when you read a character that ends the current token (e.g. a space after a number), you want to push it back so the *next* call to the tokenizer can see it. If that "next character" happens to be EOF, you'd like to push it back too - and read EOF again next time.

The single-sentinel design from ex-4-8 silently *drops* a pushed-back EOF: pushing back `EOF` is equivalent to "now the buffer is empty", so the next `getch` will call `getchar` again, which may or may not return EOF depending on whether the stream has actually ended.

The flag-based design preserves the invariant "what you push back is what you get back next".

## Modern note

A more general solution is a small ring buffer (the original §4 buffer) that holds N pushbacks regardless of value. For complex parsers (Pratt parsers, infinite look-ahead), libraries like `flex` or `ANTLR` handle this entirely.

For most C programs the flag-based single-pushback above is sufficient. It's the minimum complexity that handles every valid input correctly.

## Notes from the author

- "Sentinel values are dangerous" is a recurring C lesson. NULL-terminated strings have the same problem (you can't put `\0` in a string). The fix is always the same: separate the *flag* from the *value*.
- The bool addition is one byte of memory and one branch in the hot path. The cost is negligible; the safety gain is real. Worth it.
- Many parsers actually want this exact contract: "I might want to ungetch EOF after a `peek` that returned EOF". Without it, you have to special-case EOF detection at every callsite. With it, the abstraction is symmetric.
