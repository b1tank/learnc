> Extend `dcl` to handle declarations with function-argument lists, type qualifiers like `const` and `volatile`, and struct/union qualifiers.

## Approach

Three additions:

1. **Type qualifiers** in the leading type word: `const int`, `volatile char`. Read them in the type loop until you see a non-qualifier.
2. **Function arguments** in the `()` after a `dirdcl`: read tokens between matching parens, accumulate them into the description.
3. **Struct tags**: `struct point` is two tokens that together form one type.

```c
typedef struct { const char *name; } qualifier;
static const char *qualifiers[] = { "const", "volatile", "static", "extern", NULL };

static int is_qualifier(const char *w) {
    for (int i = 0; qualifiers[i]; ++i)
        if (strcmp(w, qualifiers[i]) == 0) return 1;
    return 0;
}
```

In the main loop, accumulate the type by reading consecutive qualifier+type tokens:

```c
char type_phrase[256] = "";
while (gettoken() == NAME && is_qualifier(token)) {
    strcat(type_phrase, token);
    strcat(type_phrase, " ");
}
if (tokentype == NAME) {              /* the base type */
    strcat(type_phrase, token);
}
```

For function arguments, when you see `(` after a name, switch to a sub-parser that consumes tokens until the matching `)`:

```c
case '(':
    strcat(out, " function (");
    int depth = 1;
    while (depth > 0) {
        gettoken();
        if (tokentype == '(')      ++depth;
        else if (tokentype == ')') --depth;
        if (depth > 0) {
            strcat(out, token);
            strcat(out, " ");
        }
    }
    strcat(out, ") returning");
    break;
```

## Notes from the author

- The full `dcl` extension is a real project — ~100-150 lines of careful tokenising and recursive descent. The skeleton above is a stub; building it out is a great exercise in incremental parser growth.
- C's declaration syntax was the inspiration for one of K&R's most quoted lines: "Almost everyone agrees that the declaration syntax is at best regrettable and at worst near-incomprehensible." `dcl` exists *because* the syntax is hard to read.
- Modern tools (`clang -fsyntax-only -fdump-tokens`, `cdecl`) handle full ISO C declarations including all qualifiers, function pointers with complex argument lists, and bitfields. Studying their source (or `cdecl`'s 2000-line C source) shows how a real declaration parser handles each corner.
