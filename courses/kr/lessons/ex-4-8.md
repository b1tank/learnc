> Suppose at most **one** character is ever pushed back. Rewrite `getch` and `ungetch` to take advantage of this - no buffer needed.

## Approach

Replace the `char buf[BUFSIZE]` + `bufp` index with a single `int` slot. Use a sentinel value (`EOF` is a natural choice, since it's `-1` and outside the valid character range) to mean "the slot is empty".

```c:starter
#include <stdio.h>

static int pushback = EOF;     /* EOF means: no character pushed back */

int getch(void) {
    if (pushback != EOF) {
        int c = pushback;
        pushback = EOF;
        return c;
    }
    return getchar();
}

void ungetch(int c) {
    if (pushback != EOF)
        fprintf(stderr, "ungetch: only one char allowed\n");
    else
        pushback = c;
}

int main(void) {
    ungetch('X');
    printf("%c\n", getch());      /* 'X' */
    printf("(buffer empty now)\n");
    ungetch('Y');
    printf("%c\n", getch());      /* 'Y' */
    return 0;
}
```

```output
X
(buffer empty now)
Y
```

## What's gained

- One fewer global (`bufp` is gone).
- No buffer overflow path; the only failure is "called `ungetch` twice in a row without `getch` in between".
- The code is short enough to be inlined; on modern compilers it's a single load + compare in the hot path.

## What's risked

- If you ever need two-character look-ahead (or more), you have to rewrite. The buffer version scales transparently.
- The sentinel choice (`EOF`) works *only* because `EOF` is `-1` and `getchar` returns values in `[0, 255]` plus `EOF`. If a future API change makes `getch` return arbitrary `int`s, the sentinel could collide. A separate `bool has_pushback` would be more defensive.

```c
static int  pushback;
static bool has_pushback = false;
```

That's cleaner; the cost is one extra byte and a slightly more verbose check.

## Notes from the author

- The "single sentinel value" pattern is everywhere in C - `NULL` for pointers, `EOF` for I/O, `(size_t)-1` for "no result" in `strstr`. It's compact but fragile; a flag-and-value pair is safer.
- One-character look-ahead is enough for most simple parsers (numeric literals, identifiers in C-like grammars). Multi-character look-ahead is needed for things like distinguishing `<<` from `<<=` from `<` - but tokenizers handle that by looking at the *next* character only, not by un-getting many.
- This is a classic "premature generalisation" trap: if you only ever need one pushback, the buffer is dead code that takes up space and obscures the intent. Match the API to the actual use case.
