> Enhance the §8.7 allocator so it gracefully handles edge cases: zero-size requests, suspiciously large requests, and integer overflow.

## Approach

Add a guarding wrapper that does sanity checks before delegating to the real allocator:

```c
void *safe_malloc(size_t n) {
    if (n == 0) return NULL;        /* or return a 1-byte buffer per impl choice */
    if (n > MAX_REASONABLE) {
        fprintf(stderr, "safe_malloc: %zu too large\n", n);
        return NULL;
    }
    void *p = real_malloc(n);
    if (!p) {
        fprintf(stderr, "safe_malloc: out of memory for %zu bytes\n", n);
        return NULL;
    }
    return p;
}
```

```c:starter
#include <stdio.h>
#include <stdlib.h>

#define MAX_REASONABLE (1ULL << 30)    /* 1 GiB */

static long alloc_count = 0;
static long alloc_bytes = 0;

void *safe_malloc(size_t n) {
    if (n == 0) {
        fprintf(stderr, "[warning] safe_malloc(0)\n");
        return NULL;
    }
    if (n > MAX_REASONABLE) {
        fprintf(stderr, "[warning] safe_malloc(%zu) > MAX_REASONABLE\n", n);
        return NULL;
    }
    void *p = malloc(n);
    if (!p) {
        fprintf(stderr, "[error] OOM for %zu bytes\n", n);
        return NULL;
    }
    ++alloc_count;
    alloc_bytes += (long)n;
    return p;
}

int main(void) {
    int  *a = safe_malloc(100 * sizeof *a);
    char *b = safe_malloc(0);                  /* warning */
    char *c = safe_malloc((size_t)-1);          /* warning */
    void *d = safe_malloc(MAX_REASONABLE / 2);

    printf("a=%p, b=%p, c=%p, d=%p\n", (void *)a, (void *)b, (void *)c, d);
    printf("alloc_count=%ld, alloc_bytes=%ld\n", alloc_count, alloc_bytes);

    free(a); free(d);
    return 0;
}
```

```output
[warning] safe_malloc(0)
[warning] safe_malloc(...) > MAX_REASONABLE
a=0x..., b=(nil), c=(nil), d=0x...
alloc_count=2, alloc_bytes=...
```

## What "reasonable" means

`MAX_REASONABLE` depends on the program:

- An embedded controller: 64 KB.
- A web server: 256 MB.
- A scientific simulation: tens of GB.

There's no universal answer; pick a number that exceeds normal use but flags clear bugs (e.g. accidentally allocating `(uint32_t)-1` bytes due to subtraction underflow).

## C standard behaviour for `malloc(0)`

The standard allows two behaviours:

- Return `NULL`.
- Return a unique pointer that can be `free`d but not dereferenced.

Both are valid! Code that calls `malloc(0)` should treat NULL as success. This is why many wrappers warn — `malloc(0)` is *probably* a bug in the caller.

## Tracking allocations

The `alloc_count`/`alloc_bytes` totals are a tiny "leak detector": at exit, you can print them and verify that the allocation count matches the free count. Production-grade tools (valgrind, AddressSanitizer) do this in extreme detail.

## Try it

1. Track free count too. Print "leaked: N bytes in M allocations" at exit.
2. Add a `--max-alloc=N` CLI flag to override `MAX_REASONABLE`.
3. Make the warnings go to a log file instead of stderr.

## Notes from the author

- The pattern "thin wrapper that logs and sanity-checks" is the cheapest debugging investment in C. A 20-line `safe_malloc`/`safe_free` pair finds many bugs.
- Real libcs already do bounds checking on debug builds (`MALLOC_CHECK_` for glibc, `MallocStackLogging` on macOS). Turn them on during development; turn them off in production.
- The "abort on big request" pattern is debatable. Some applications legitimately allocate gigabytes. Others use `1ULL << 30` accidentally and need to crash early. Know your domain.
