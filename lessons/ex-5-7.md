> The §5.6 `readlines` allocates each line with a separate `malloc` and keeps an array of pointers. Rewrite it to store all line text in a single big buffer, with the pointer array pointing into that buffer.

## Approach

A two-array layout:

- `char text[BIG]` — one large char buffer with all lines concatenated, each terminated by `'\0'`.
- `char *lineptr[MAX]` — array of pointers into `text`.

When you read line N, write its bytes to the current end of `text`, set `lineptr[N]` to point there, append a `'\0'`, and advance the write position.

```c:starter
#include <stdio.h>
#include <string.h>

#define MAXLINES 100
#define ALLOCSIZE 5000

static char  allocbuf[ALLOCSIZE];
static int   allocp = 0;
static char *lineptr[MAXLINES];

/* return pointer to n bytes inside the big buffer, or NULL */
static char *alloc_in_buf(int n) {
    if (ALLOCSIZE - allocp < n)
        return NULL;
    char *p = &allocbuf[allocp];
    allocp += n;
    return p;
}

int read_lines(char *lineptr[], int maxlines) {
    int nlines = 0;
    char buf[1024];
    while (nlines < maxlines && fgets(buf, sizeof buf, stdin)) {
        int n = (int)strlen(buf);
        char *slot = alloc_in_buf(n + 1);
        if (!slot) break;
        memcpy(slot, buf, n + 1);
        lineptr[nlines++] = slot;
    }
    return nlines;
}

int main(void) {
    int nlines = read_lines(lineptr, MAXLINES);
    for (int i = 0; i < nlines; ++i)
        printf("L%2d: %s", i, lineptr[i]);
    return 0;
}
```

## What changes

| Property                       | Per-line malloc                  | Single buffer                  |
|--------------------------------|----------------------------------|--------------------------------|
| Number of allocations          | One per line                     | Zero (static) or one (heap)    |
| Per-line overhead              | ~16-32 bytes for malloc bookkeeping | Zero                       |
| Cache locality                 | Pointer-chasing                  | Sequential — fast scans        |
| Free                           | Must walk every pointer           | Free the whole buffer once     |
| Capacity                       | Limited by total RAM             | Limited by buffer size         |

For "read once, scan many times", the single-buffer layout is much faster — every line is in the same cache lines.

## Notes from the author

- The single-buffer-with-pointers layout is what real libraries (e.g. Perl's stringbuf, NSS's NSPR strings) call a "string heap". It's `O(1)` per line after the initial allocation; pointer-chasing through `malloc`'d lines is `O(log n)` per access on a fragmented heap.
- The `alloc_in_buf` helper is a tiny bump allocator — increment a counter, never free individual entries. This is the simplest possible allocator and the right tool when you want all your data to die together.
- The trade-off: you can't `free` one line in the middle. But you usually don't want to — these line collections are typically built up once and discarded together.
