> Pick the array-based functions from earlier chapters — `strcpy`, `atoi`, `itoa`, `reverse`, etc. — and rewrite them with pointers instead of subscripts. Which version do you prefer?

## Approach

The translation is mechanical: replace `s[i]` with `*p` (where `p` walks the string), replace `++i` with `++p`. Loops that iterate over a string take advantage of `*p++` and `while (*p)` patterns.

Here's `reverse` from §3.6 in pointer style:

```c:starter
#include <stdio.h>
#include <string.h>

void reverse_p(char *s);

int main(void) {
    char a[] = "hello world";
    reverse_p(a);
    printf("[%s]\n", a);
    return 0;
}

void reverse_p(char *s) {
    char *end = s + strlen(s) - 1;
    while (s < end) {
        char t = *s;
        *s++ = *end;
        *end-- = t;
    }
}
```

And `atoi` in pointer style:

```c
int atoi_p(const char *s) {
    while (isspace((unsigned char)*s)) ++s;
    int sign = 1;
    if (*s == '+' || *s == '-')
        sign = (*s++ == '+') ? 1 : -1;
    int n = 0;
    while (isdigit((unsigned char)*s))
        n = 10*n + (*s++ - '0');
    return sign * n;
}
```

## Which is preferable?

| Subscript form (`s[i]`)        | Pointer form (`*p`, `*p++`)         |
|--------------------------------|--------------------------------------|
| Index is visible — easy debug   | Pointer walk is denser                |
| Random access (any `s[i]`)      | Sequential walk only                  |
| Index `i` for error messages    | "Offset from where?" is trickier       |
| Familiar to non-C programmers   | Idiomatic K&R-era C                    |

Modern compilers generate **identical code** for both at `-O2`. So the choice is purely stylistic:

- Pointer form for tight inner loops and library-style helpers.
- Subscript form when the loop variable is meaningful (e.g. "process every 4th byte" → `s[4*i]`).

I tend to prefer subscripts for new code: debuggers print the index, and it's easier to read aloud ("`s[i]` is the i-th character"). K&R-era books overdo the pointer style; modern style guides are mixed.

## Notes from the author

- The pointer-walk style was *necessary* in 1978 — early C compilers didn't optimise indexing well, and pointer code measurably ran faster. Today the optimisers see through both forms and the runtime is identical.
- The translation from indices to pointers is rote: `s[i]` → `p[0]` (where `p` walks); `s[i+1]` → `p[1]`. The semantics don't change.
- One real reason to prefer pointers: when you don't have a base array, just a "current position" — e.g. parsing a chunk of bytes you got from a network read. The pointer is the natural representation; the index would require carrying the base around.
