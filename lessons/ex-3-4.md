> The `itoa` from §3.6 fails on the most-negative integer (`INT_MIN`, typically `-2147483648` on 32-bit `int`). The bug: `n = -n` overflows for `INT_MIN` because the positive form `+2147483648` is *not* representable in a signed 32-bit int. Rewrite `itoa` so it handles this case correctly.

## Approach

The trick: do the **modulo and division work on the negative side**, then flip signs *only when emitting the digit character*. Negative numbers have a representable magnitude that's exactly one larger than positive ones in two's complement, so working with the negative form is always safe.

```c
int sign = (n < 0) ? -1 : 1;
do {
    int digit = (n < 0) ? -(n % 10) : (n % 10);
    s[i++] = digit + '0';
    n /= 10;
} while (n != 0);
```

Some platforms have implementation-defined behaviour for `%` with negatives (pre-C99), so an even more portable trick is:

```c
int digit = sign * (n % 10);   /* sign==-1 flips it positive */
```

works the same on C99+ where `%` truncates toward zero (so `-7 % 10 == -7`).

```c:starter
#include <stdio.h>
#include <limits.h>
#include <string.h>

void itoa(int n, char s[]);
void reverse(char s[]);

int main(void) {
    char buf[32];
    itoa(0,        buf); printf("itoa(0)        = %s\n", buf);
    itoa(42,       buf); printf("itoa(42)       = %s\n", buf);
    itoa(-1,       buf); printf("itoa(-1)       = %s\n", buf);
    itoa(INT_MAX,  buf); printf("itoa(INT_MAX)  = %s\n", buf);
    itoa(INT_MIN,  buf); printf("itoa(INT_MIN)  = %s\n", buf);   /* the tricky one */
    return 0;
}
```

## Solution

```c
void itoa(int n, char s[]) {
    int sign = n;            /* remember sign before mutation */
    int i = 0;

    do {
        /* digit is non-negative regardless of n's sign */
        int digit = n % 10;
        if (digit < 0) digit = -digit;
        s[i++] = digit + '0';
        n /= 10;
    } while (n != 0);

    if (sign < 0)
        s[i++] = '-';

    s[i] = '\0';
    reverse(s);
}

void reverse(char s[]) {
    int i = 0, j = (int)strlen(s) - 1;
    while (i < j) {
        char t = s[i]; s[i] = s[j]; s[j] = t;
        ++i; --j;
    }
}
```

```output
itoa(0)        = 0
itoa(42)       = 42
itoa(-1)       = -1
itoa(INT_MAX)  = 2147483647
itoa(INT_MIN)  = -2147483648
```

## Notes from the author

- The "never make INT_MIN positive" rule applies to many algorithms. `abs(INT_MIN)` is undefined behaviour, `-INT_MIN` overflows. The defensive habit: work with the negative form, only flip signs when *outputting*.
- The same problem appears in division: `INT_MIN / -1` overflows on some platforms because the result `+2147483648` isn't representable. Compilers warn about this with `-ftrapv`.
- Modern alternative: use `snprintf(buf, sizeof buf, "%d", n)`. It's correct, fast, locale-aware. The point of `itoa` is to learn how integer-to-string conversion *works*, not to replace `snprintf`.
