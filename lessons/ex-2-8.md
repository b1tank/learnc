> **Exercise 2-8.** Write `rightrot(x, n)` that returns `x` rotated right by `n` bit positions. Bits that fall off the right end appear on the left.

## Approach

A rotation is two shifts glued together:

- The bottom `n` bits of `x` (the ones that "wrap around") go to the top.
- The remaining bits shift right.

If `w` is the word width, the result is `(x >> n) | (x << (w - n))`. Mask `n` modulo `w` first so callers can rotate by any amount.

```c:starter
#include <stdio.h>
#include <limits.h>

unsigned rightrot(unsigned x, int n);

int main(void) {
    /* 0xF0000001 rotated right by 4 = 0x1F000000 */
    printf("%08X\n", rightrot(0xF0000001U, 4));
    printf("%08X\n", rightrot(0x12345678U, 8));   /* expect 78123456 */
    return 0;
}

unsigned rightrot(unsigned x, int n) {
    /* TODO */
    return x;
}
```

## Solution

```c
#include <limits.h>

unsigned rightrot(unsigned x, int n) {
    const int width = sizeof(unsigned) * CHAR_BIT;
    n %= width;                                /* normalise large n */
    if (n == 0) return x;                      /* avoid x << width UB */
    return (x >> n) | (x << (width - n));
}
```

```output
1F000000
78123456
```

## Notes from the author

- The `if (n == 0) return x;` is not just an optimisation — without it, `x << (width - 0) == x << 32` on a 32-bit `unsigned` is **undefined behaviour**. Shift-by-width is a real gotcha.
- A common alternative formulation: peel `n` low bits, shift the rest, OR them back. Slightly more readable, identical machine code.
- Modern x86 has a single instruction for rotate (`ROR`). Most compilers recognise the `(x >> n) | (x << (w - n))` idiom and emit it. So you get the readable expression *and* the optimal machine code — a happy alignment.
