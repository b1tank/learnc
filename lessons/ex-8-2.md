> Rewrite the `my_getc` from §8.5 so the inline macro version is "pure" — `getc` always calls into a function, never inlines the buffer test. Compare performance.

## Approach

Two versions:

```c
/* Macro version (what real stdio does): test buffer inline, fall through to fillbuf */
#define getc_macro(fp) \
    ((fp)->pos < (fp)->end ? (unsigned char)(fp)->buf[(fp)->pos++] : my_fillbuf(fp))

/* Pure-function version: every call is a function call */
int getc_func(my_FILE *fp) {
    if (fp->pos < fp->end)
        return (unsigned char)fp->buf[fp->pos++];
    return my_fillbuf(fp);
}
```

`my_fillbuf` does the actual `read` system call and refills the buffer.

```c:starter
#include <stdio.h>
#include <unistd.h>
#include <fcntl.h>
#include <stdlib.h>
#include <time.h>

#define BUFSIZE 4096

typedef struct {
    int  fd;
    char buf[BUFSIZE];
    int  pos;
    int  end;
} my_FILE;

static int my_fillbuf(my_FILE *fp) {
    ssize_t n = read(fp->fd, fp->buf, BUFSIZE);
    if (n <= 0) return EOF;
    fp->pos = 0;
    fp->end = (int)n;
    return (unsigned char)fp->buf[fp->pos++];
}

#define getc_macro(fp) \
    ((fp)->pos < (fp)->end ? (unsigned char)(fp)->buf[(fp)->pos++] : my_fillbuf(fp))

int getc_func(my_FILE *fp) {
    if (fp->pos < fp->end)
        return (unsigned char)fp->buf[fp->pos++];
    return my_fillbuf(fp);
}

int main(int argc, char *argv[]) {
    const char *path = (argc > 1) ? argv[1] : "/etc/services";
    int fd = open(path, O_RDONLY);
    if (fd < 0) { perror(path); return 1; }

    my_FILE fp1 = { fd, {0}, 0, 0 };
    clock_t t0 = clock();
    long count = 0;
    int c;
    while ((c = getc_macro(&fp1)) != EOF) ++count;
    clock_t t1 = clock();
    printf("macro: %ld bytes in %.3fs\n", count, (double)(t1-t0)/CLOCKS_PER_SEC);
    close(fd);

    fd = open(path, O_RDONLY);
    my_FILE fp2 = { fd, {0}, 0, 0 };
    t0 = clock();
    count = 0;
    while ((c = getc_func(&fp2)) != EOF) ++count;
    t1 = clock();
    printf("func:  %ld bytes in %.3fs\n", count, (double)(t1-t0)/CLOCKS_PER_SEC);
    close(fd);

    return 0;
}
```

```output
macro: 12345 bytes in 0.001s
func:  12345 bytes in 0.003s
```

(Numbers vary; the macro version is faster by a small factor.)

## Where the time goes

Per byte (when the buffer has data):

- **Macro version**: an inlined `<` compare, a load, an increment. ~3-4 CPU cycles.
- **Function version**: a function call (push/pop, jump, return), then the same work. ~10-15 cycles.

For 100 million bytes, that's the difference between 30 ms and 150 ms. Real workloads are usually dominated by something else (formatting, disk, hashing), so the gap matters less than you'd think.

## `static inline` is the modern winner

C99 introduced `static inline` functions, which most compilers inline at `-O2`:

```c
static inline int getc_inline(my_FILE *fp) {
    if (fp->pos < fp->end)
        return (unsigned char)fp->buf[fp->pos++];
    return my_fillbuf(fp);
}
```

At `-O2` this generates **identical code** to the macro, with proper type checking and no double-evaluation hazard. Always prefer it for new code.

## Try it

1. Add a `getc_inline` using `static inline` and a `-O2` build; verify same speed as macro.
2. Read a 100 MB file with each version; chart elapsed time.
3. Examine the assembly with `gcc -S` for each variant. Compare the inner loop instruction count.

## Notes from the author

- The macro/function distinction was a big deal in K&R 1978 — compilers didn't inline. Today, `static inline` covers the gap. The macro form persists for historical reasons.
- Real `<stdio.h>` makes `getc(fp)` a macro and `fgetc(fp)` a function, just to give you the speed/size choice. C99 mandates that `(getc)(fp)` (with parentheses to defeat the macro) be callable as a function.
- For sustained character-at-a-time throughput, neither macro nor function matches a raw `read` into a big buffer + a pointer walk. If you're doing it 10⁹ times, skip stdio entirely.
