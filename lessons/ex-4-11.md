> Modify `getop` so it doesn't need `ungetch`. Hint: use an internal `static` variable to remember the look-ahead character between calls.

## Approach

The trick is to keep, *inside* `getop` itself, a `static int` holding "the character we already read but haven't classified yet". On entry, if that slot is set, consume it first; otherwise call `getchar`. When the function ends with a character that wasn't part of the current token, store it in the slot instead of calling `ungetch`.

```c:starter
#include <stdio.h>
#include <ctype.h>
#include <stdlib.h>

#define NUMBER '0'
#define MAXOP  20

/* returns the operator type or a NUMBER token */
int getop(char s[]) {
    static int last = ' ';   /* peek slot: ' ' is "no character buffered" */

    int c;
    /* take from peek slot if present, else read fresh */
    if (last != ' ') {
        c = last;
        last = ' ';
    } else {
        c = getchar();
    }

    while (c == ' ' || c == '\t')
        c = getchar();

    s[0] = (char)c;
    s[1] = '\0';
    if (!isdigit(c) && c != '.')
        return c;                 /* operator or EOF */

    int i = 0;
    if (isdigit(c))
        while (isdigit(s[++i] = (char)(c = getchar())))
            ;
    if (c == '.')
        while (isdigit(s[++i] = (char)(c = getchar())))
            ;
    s[i] = '\0';

    /* c is the first non-numeric character; stash it for next call */
    if (c != EOF)
        last = c;

    return NUMBER;
}

int main(void) {
    char tok[MAXOP];
    int  t;
    while ((t = getop(tok)) != EOF) {
        if (t == NUMBER) printf("NUMBER: %s\n", tok);
        else             printf("OPERATOR: %c\n", t);
    }
    return 0;
}
```

```output
(awaits input — try '12 34 + 5 *')
```

## What's different

Before: a separate `ungetch` function and an external `buf`/`bufp` pair, used by every consumer of `getop`.

After: `getop` is self-contained. The look-ahead state is *encapsulated* in a `static` local — only `getop` knows about it, no other function can mess with it.

## The downside

- The pattern only works because `getop` is the **only consumer** of `getchar`. If another part of the program also reads `stdin`, the cached character is invisible to it.
- Re-entrancy is broken: if `getop` somehow calls itself (it doesn't here, but consider future maintenance), the static slot would lose data.

For a single-purpose tokenizer, this is fine. For a general lexer used by multiple parsers, an explicit state struct passed by pointer is better:

```c
typedef struct { int peek; bool has_peek; } Lexer;
int getop_r(Lexer *L, char s[]);
```

The `_r` suffix is the C convention for "re-entrant" / "thread-safe" versions of stateful functions (`strtok_r`, `localtime_r`).

## Notes from the author

- The "static local for state hidden inside one function" pattern is the *exact* situation where C99's `static` inside a function shines. We get encapsulation without a struct.
- The sentinel `' '` for "empty peek slot" has the same drawback as `EOF` in ex-4-8: it can't be a valid input. Whitespace is skipped here so it's safe, but a separate `bool has_peek` is more defensive.
- "Make it stateless or make state explicit" is the modern lesson. Hidden globals (including `static` locals) make code hard to test and impossible to reuse in multi-threaded contexts. For teaching purposes the pattern is fine; in production, prefer the explicit struct.
