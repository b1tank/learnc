> **Exercise 1-6.** Convince yourself by experiment that the expression `getchar() != EOF` evaluates to `0` or `1`.

## Approach

In C, every relational and equality operator produces an `int` with value `0` (false) or `1` (true). That includes `!=`. So `getchar() != EOF` is always one of those two integers, no matter what character came in.

Print the value of the expression on every iteration. While stdin still has input, you'll see `1`. When the stream ends and the next `getchar()` returns `EOF`, the same expression yields `0` — and the loop exits.

```c:starter
#include <stdio.h>

int main(void) {
    /* TODO: print 1 for each character read, 0 when stream ends */
    return 0;
}
```

## Solution

```c
#include <stdio.h>

int main(void) {
    int v;

    /* loop until EOF, printing the value of the comparison each time */
    while ((v = (getchar() != EOF)) != 0)
        printf("v = %d\n", v);

    printf("v = %d (loop exited)\n", v);
    return 0;
}
```

```output
v = 1
v = 1
v = 1
v = 1
v = 0 (loop exited)
```

For four input characters followed by EOF.

## Notes from the author

- The point of this exercise — usually skipped by experienced programmers — is to internalise that **C's booleans are integers**. `1` and `0` aren't just shorthand; they *are* the values, and you can do arithmetic on them. `int n = (x > 0) + (y > 0)` counts how many of `x` and `y` are positive.
- This is the trick behind the famously dense `(c = getchar()) != EOF` idiom: the whole expression *is* the loop condition because it has a numeric truthiness.
- C99 added `<stdbool.h>` with `bool`, `true`, `false` — typedefs over `int`. They're nicer to read but mechanically identical. Pre-C99 code (including K&R itself) uses bare `int`.
