> Enhance the §6.4 keyword counter so it correctly skips contents of string literals, character constants, single-line `//` comments, multi-line `/* */` comments, and `#` preprocessor lines. Words inside those should NOT be counted as identifiers.

## Approach

A small state machine. Read a character at a time. The state tells you whether you're in normal code, inside a string, inside a `/*` comment, etc. While inside a non-code state, do nothing with the characters except watch for the terminating sequence.

| State          | Stay until                | Then go to        |
|----------------|---------------------------|-------------------|
| NORMAL         | `"` → STRING              | STRING             |
| NORMAL         | `'` → CHARCONST           | CHARCONST          |
| NORMAL         | `/*` → BLOCK_COMMENT      | BLOCK_COMMENT      |
| NORMAL         | `//` → LINE_COMMENT       | LINE_COMMENT       |
| NORMAL         | `#` at line start → PP    | PP                 |
| STRING         | unescaped `"`             | NORMAL             |
| CHARCONST      | unescaped `'`             | NORMAL             |
| BLOCK_COMMENT  | `*/`                      | NORMAL             |
| LINE_COMMENT   | `\n`                      | NORMAL             |
| PP             | `\n` (not preceded by `\`)| NORMAL             |

```c:starter
#include <stdio.h>
#include <ctype.h>

enum state { S_NORMAL, S_STRING, S_CHAR, S_BLOCK, S_LINE, S_PP };

int main(void) {
    int c, prev = '\n';
    enum state s = S_NORMAL;
    int word_count = 0;
    int in_word = 0;

    while ((c = getchar()) != EOF) {
        switch (s) {
        case S_NORMAL:
            if (c == '"')                       s = S_STRING;
            else if (c == '\'')                 s = S_CHAR;
            else if (c == '/' && (c = getchar()) == '*') s = S_BLOCK;
            else if (c == '/' && c == '/')      s = S_LINE;
            else if (c == '#' && prev == '\n')  s = S_PP;
            else {
                if (isalpha(c) || c == '_') {
                    if (!in_word) { ++word_count; in_word = 1; }
                } else if (!isalnum(c) && c != '_') {
                    in_word = 0;
                }
            }
            break;
        case S_STRING:
            if (c == '\\') getchar();           /* skip escape */
            else if (c == '"') s = S_NORMAL;
            break;
        case S_CHAR:
            if (c == '\\') getchar();
            else if (c == '\'') s = S_NORMAL;
            break;
        case S_BLOCK:
            if (c == '*' && (c = getchar()) == '/') s = S_NORMAL;
            break;
        case S_LINE:
            if (c == '\n') s = S_NORMAL;
            break;
        case S_PP:
            if (c == '\n' && prev != '\\') s = S_NORMAL;
            break;
        }
        prev = c;
    }
    printf("identifier-like tokens: %d\n", word_count);
    return 0;
}
```

## Why state machines

This is the textbook example of code that benefits from explicit state. Without it you'd end up with deeply nested `if`s and one-off flag variables. With a `state` variable and a `switch`, each transition is visible and easy to extend (add a new state for raw string literals, for example).

## Try it

2. Add raw-string support (C++ has `R"(...)"`; C doesn't, but C-with-extensions sometimes does).
3. Count lines and produce a histogram of identifier lengths.

## Notes from the author

- Lexers are state machines. Once you internalise the pattern, you can write a lexer for any small language in an hour. The transitions ARE the spec.
- Real lexers use generated tools (lex, flex, re2c) for performance and correctness. Hand-written ones are common for "small enough" languages and for performance-critical paths in compilers.
- Comments inside strings and strings inside comments are the classic edge cases. State machines handle them without special-casing.
